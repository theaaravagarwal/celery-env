Celery gives app config the ergonomics of a schema and the startup profile of
plain hand-written code. Author one env schema, generate a validator, and ship a
self-contained module with no runtime package import on the hot boot path.

```sh
npm install -D celery-env
npx celery-env init --target node --schema env.schema.mjs
npx celery-env generate --schema env.schema.mjs --out src/env.mjs --types src/env.d.ts --example .env.example --minify
```

```js
// src/config.js
import { loadEnv } from "./env.mjs";

export const env = loadEnv(process.env);
```

## Why Celery

- **Generated by default**: no schema walk, no parser framework, no runtime dependency import.
- **Tiny output**: the measured small generated validator is 526 gzip bytes.
- **Real env semantics**: defaults, `devDefault`, `testDefault`, `requiredWhen`, URL protocols, strict numbers, lists, JSON, and secret-safe aggregate errors.
- **Runtime fallback**: use `parseEnv()` directly when a build step is not worth it.
- **Interop path**: export JSON Schema metadata for docs, forms, and ecosystem tooling.
- **Auditable release shape**: zero root dependencies, fixed package-file whitelist, hardened generated code, and CI-backed security checks.

## Example

```js
// env.schema.mjs
import { bool, defineEnv, int, oneOf, str, url } from "celery-env";

export default defineEnv({
  NODE_ENV: oneOf(["development", "test", "production"], { default: "development" }),
  DATABASE_URL: url({ protocols: ["postgres:"] }),
  PORT: int({ default: 3000, min: 1, max: 65535 }),
  ENABLE_SIGNUPS: bool({ default: false }),
  SESSION_SECRET: str({
    optional: true,
    requiredWhen: (env) => env.NODE_ENV === "production",
    desc: "Required in production."
  })
});
```

```sh
npx celery-env generate --schema env.schema.mjs --out src/env.mjs --types src/env.d.ts
```

```js
import { loadEnv } from "./env.mjs";

const env = loadEnv(process.env);
console.log(env.PORT);
```

For existing apps with nested config, keep the generated validator flat and put
only the adapter in maintained code:

```js
import { loadEnv } from "./env.mjs";

export function loadConfig(source = process.env) {
  const env = loadEnv(source);
  return {
    app: { nodeEnv: env.NODE_ENV, port: env.PORT },
    database: { url: env.DATABASE_URL }
  };
}
```

## Runtime API

```js
import { defineEnv, int, parseEnv, str } from "celery-env";

const schema = defineEnv({
  DATABASE_URL: str({ min: 1 }),
  PORT: int({ default: 3000 })
});

const env = parseEnv(schema, process.env);
```

## Ecosystem Interop

```js
import { generateJsonSchema } from "celery-env/compiler";
import schema from "./env.schema.mjs";

export const jsonSchema = generateJsonSchema(schema, {
  title: "Orders API environment"
});
```

Celery-specific behavior is preserved as `x-celery-*` metadata so tooling can
consume the schema without putting a general-purpose validator in the generated
startup path.

## Benchmarks

Current local report: Node v26.3.0, macOS arm64, Apple M3.

| Metric | Result |
| --- | ---: |
| Valid real-schema geometric mean | 1.50x over best external competitor |
| Real `process.env` geometric mean | 1.14x over best external competitor |
| Invalid real-schema geometric mean | 1.32x over best external competitor |
| Cold first validation | 2.42x faster than best external competitor |
| Generated bundle size | 526 gzip bytes |
| Smallest external bundle gap | 2.15x smaller |

Competitors measured include Zod, Valibot, Envalid, Envsafe, env-var, T3 Env
Core, Valienv, env-schema, env-type-validator, safe-env-vars, and Convict where
the benchmark applies. The claim is specific: Celery wins this env-validation
corpus; it is not a blanket statement about every validation workload.

```sh
cd sandbox/bench
npm run report
```

The full report is written to `sandbox/bench/artifacts/report.md` and
`sandbox/bench/artifacts/report.json`.

## Security

The published package has zero dependencies, no consumer install hooks, and a
fixed package-file whitelist. Runtime and generated validators avoid
rejected-value leakage, use own-property env reads, escape generated schema keys,
validate generated function names, and preserve `__proto__` keys as data
properties.

```sh
npm run security:scan
npm run ci
```

See [SECURITY.md](SECURITY.md) for the trust boundary and hardening notes.
